# AI_INSTRUCTIONS.md

Инструкция для AI-ассистента (Qwen 3.5, Claude, GPT и т. п.), которого позвали помочь с интеграцией VisionSystem-фронта в реальный бекенд.

## 1. Контекст проекта

VisionSystem — CV-сервис для производства: контроль зон безопасности, проверка спецодежды (PPE), интеграция с Trassir NVR, журнал событий. Этот репозиторий — фронт на React + TypeScript + Vite, который работает поверх уже существующего Python-бекенда (API на порту `:8080`). Бекенд задокументирован автором сервиса отдельно, этот репозиторий его не содержит. Автор фронта — Николай Проклов (NikolasPikchers), заказчик — Русклимат.

## 2. Твоя задача (для AI)

- **Изучи** сначала `README.md` в корне и структуру `src/`, прежде чем что-то писать. Не исходи из догадок — код уже есть.
- **Не переписывай фронт.** Он готов, типизирован, покрыт тестами (Vitest, 28 штук). Твоя задача — помочь подружить его с реальным API и починить мелкие расхождения (формат лог-строк, отсутствующие CORS-заголовки, и т. п.).
- **Перед любым изменением** спроси пользователя: на чём у него бекенд (Python/Go/Node/…), на каком порту, как он планирует раздавать статику (рядом с API, отдельный nginx, другое). Без этого легко полезть править не туда.

## 3. Чек-лист интеграции (выполняй по порядку, отчитывайся после каждого шага)

1. **Поставить зависимости и проверить dev-режим.**
   ```bash
   npm install
   npm run dev
   ```
   Dev-сервер должен стартовать на `http://localhost:5173`. Если нет — читай stdout, чаще всего проблема в несовпадении версии Node (нужна ≥ 20).

2. **Настроить окружение.**
   ```bash
   cp .env.example .env
   ```
   Отредактируй `.env`, выстави `VITE_API_BASE_URL` на реальный адрес бекенда (например `http://192.168.1.10:8080`).

3. **Проверить логин.** Открой `/login`, введи `admin/admin` (или реальные креды). В ответ должен прийти `session_id` — можно убедиться через DevTools → Network → `/api/login`. После логина браузер редиректит на `/cameras`.

4. **Проверить камеры.** На `/cameras` должны подгрузиться плитки с превью. Превью — это фоновые запросы к `/api/camera/{id}/thumbnail`. Если превью пустые/битые, посмотри Network: 404 — неверный ID, 401 — сессия не передана, 5xx — смотреть логи бекенда.

5. **Проверить журнал.** На `/journal` должны парситься строки логов. Ожидаемый формат:
   ```
   [YYYY-MM-DD HH:MM:SS] [LEVEL] Camera N: message
   ```
   Если у твоего бекенда формат другой — **адаптируй регэкспы ТОЛЬКО** в `src/lib/log-parser.ts`, не трогая UI и типы. В этом файле рядом лежат Vitest-тесты, сначала поправь их, потом код.

6. **Проверить зоны.** `/admin → Зоны` → выбери камеру → нарисуй полигон → сохрани → перезагрузи страницу. Полигон должен вернуться в том же месте. Если координаты «едут» при ресайзе — значит бекенд не отдаёт `X-Frame-Width` / `X-Frame-Height` при `GET /api/zone-frame`, либо отдаёт, но заголовки скрыты CORS. Починить можно только на бекенде: добавить `Access-Control-Expose-Headers: X-Frame-Width, X-Frame-Height`.

7. **Собрать прод и подключить.** `npm run build` → получаешь `dist/`. Подключи к бекенду по одному из трёх вариантов в `README.md` (рядом с API, nginx, http.server). Убедись, что `/api/*` ходит куда надо.

## 5. Правила модификации

- Не ломай TypeScript-типы в `src/types/api.ts` — они являются источником истины для всего кода. Если формат поля на бекенде правда другой — меняй тип и синхронно правь места использования (TypeScript ругнётся и покажет где).
- Не меняй дизайн-систему (цвета/радиусы/шрифты в `tailwind.config.ts` и `src/index.css`) без явной просьбы пользователя. Эти значения — часть брендинга.
- Для нового эндпоинта API соблюдай порядок:
  1. Сначала DTO-тип в `src/types/api.ts`.
  2. Потом функция в `src/api/<resource>.ts` (либо новый файл, если ресурс новый).
  3. Если нужен кэш/ревалидация — хук `useXxx` в `src/hooks/` поверх TanStack Query.
  4. Только после этого — использование в странице.
- Все UI-строки — на русском. Ключи объектов, имена функций, переменных — на английском. Это принцип всего проекта, не нарушай его.
- Если правишь `src/lib/log-parser.ts` или `src/lib/polygon.ts` — рядом лежат Vitest-тесты. Допиши/поправь тест до кода (TDD), запусти `npm run test` и убедись, что зелёно.
- **Перед любым коммитом** прогони три команды:
  ```bash
  npm run typecheck
  npm run test
  npm run build
  ```
  Всё должно быть зелёным. Если хоть что-то красное — не коммить, разбирайся.
- Коммиты — Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`). Один коммит — одна логическая единица.

## 6. Типичные проблемы и решения

- **CORS-ошибки в консоли.** Бекенд не отдаёт `Access-Control-Allow-Origin` с нужным origin. Либо поправь на бекенде (`Access-Control-Allow-Origin: <origin фронта>`, плюс `Access-Control-Allow-Credentials: true` если есть cookie), либо разверни фронт и API на одном origin — тогда CORS не нужен вообще.
- **`session_id` истёк → редирект на `/login?reason=expired`.** Это by design: сессия живёт 30 минут, по 401 фронт сам выкидывает пользователя. Нужно продлевать без релогина — проси бекенд добавить `POST /api/refresh`.
- **Превью камер не обновляются.** К URL подставляется `?t=<ts>` автоматически, так что проблема почти всегда в прокси/nginx. Добавь `Cache-Control: no-store` на `/api/camera/*/frame` и `/api/camera/*/thumbnail`.
- **Полигоны «прыгают» при ресайзе окна или при загрузке.** Бекенд должен вернуть реальные размеры кадра в заголовках `X-Frame-Width` и `X-Frame-Height`. Если они идут, но фронт их не видит — проверь на бекенде `Access-Control-Expose-Headers: X-Frame-Width, X-Frame-Height`.
- **Логи пустые.** Проверь `/api/logs` напрямую:
  ```bash
  curl -H 'Authorization: <session_id>' http://<host>:8080/api/logs
  ```
  Если бекенд возвращает текст, а фронт показывает пусто — формат строк не совпадает с регэкспом в `src/lib/log-parser.ts`. Адаптируй.
- **Экспорт скачивает пустой файл / не скачивает вообще.** Сейчас API только кладёт файл на диск сервера и возвращает путь, браузер сам ничего не качает. Если нужен прямой download — попроси добавить `GET /api/export/download/{filename}` с `Content-Disposition: attachment` и подключи его в `src/api/export.ts`.
- **401 сразу после успешного логина.** Почти всегда — заголовок `Authorization: <session_id>` не долетает. Проверь DevTools → Network → любой запрос после логина → Request Headers. В `localStorage` под ключом `vs_auth` должен быть JSON вида `{"state":{"sessionId":"..."}}`. Если там пусто — `src/stores/auth.ts` не отработал persist, смотри консоль на ошибки.

## 7. Запрещено без подтверждения пользователя

- Не меняй `VITE_API_BASE_URL` прямо в коде. Только через `.env`.
- Не коммить секреты: реальные пароли RTSP-камер, IP Trassir-серверов, API-ключи. Если случайно попало — сотри коммит (`git reset`) и поговори с пользователем.
- Не включай `ppe_check_enabled: true` массово и сразу на всех камерах. PPE-детекция тяжёлая, может положить GPU бекенда. Включать по одной и смотреть на загрузку.
- Не переделывай дизайн: не меняй цвета, шрифты, радиусы, отступы. Дизайн — часть продукта, трогать можно только по явному запросу.
- Не удаляй MSW-моки из `src/mocks/`. Они полезны для работы без реального API (например, когда бекенд временно недоступен, либо для демо). Оставь как есть.

## 8. Перед тем как лезть в код — проверь

- Файл `.env` существует, `VITE_API_BASE_URL` выставлен и доступен из твоей сети.
- API живой:
  ```bash
  curl http://<host>:8080/api/status
  ```
  Должен вернуть что-то валидное (как минимум 200).
- Логин работает:
  ```bash
  curl -X POST http://<host>:8080/api/login \
    -H 'Content-Type: application/json' \
    -d '{"username":"admin","password":"admin"}'
  ```
  Ожидаемый ответ содержит `session_id`.
- Только после этого лезь в код. **Большинство проблем решается на уровне CORS / `.env` / сетевого доступа, а не в коде фронта.** Если в консоли браузера красные ошибки, а `curl` тоже не работает — код тут ни при чём, это инфраструктура.

## Если MSW-моки включены (`VITE_USE_MOCKS=true`)

- **Вход:** любой логин/пароль; `admin/admin` гарантированно работает.
- **Данные:** 3 камеры, 4 Trassir-канала, около 80 записей лога. Всё спрятано в `src/mocks/fixtures.ts` — можно подменить на свои данные для демо.
- **Удобно для:** первоначального знакомства с интерфейсом, демонстраций заказчику, разработки, когда реальный бекенд недоступен. В проде моки никогда не включаются (`VITE_USE_MOCKS=false`, так в `.env.example`).
- **Не работает в режиме моков:** реальное скачивание экспорта (возвращается фейковый путь), обновление превью по реальному времени (картинки статичные).
